Phase 6: Persistence (SQLite save/load)

GOAL
Implement Phase 6 from PLAN.md: SQLite schema for game persistence and REST API for save/load/resume.

COMPLETED

Step 6.1: SQLite Schema
- Created packages/server/src/database.ts — GameDatabase class
  - better-sqlite3 with WAL journal mode
  - games table: id TEXT PK, phase TEXT, turn INTEGER, state TEXT (JSON), created_at, updated_at
  - saveGame() uses UPSERT (INSERT ... ON CONFLICT DO UPDATE)
  - loadGame() deserializes JSON back to full GameState
  - listGames() returns summaries sorted by updated_at DESC
  - deleteGame() removes saved game
  - Auto-creates data/ directory via mkdirSync recursive
  - Constructor accepts optional dbPath (defaults to data/empire.db)

Step 6.2: Game Save/Load API
- Updated GameManager to accept optional GameDatabase in constructor
  - persistGame() private helper — saves game to DB (no-op if no DB)
  - Autosave after each turn execution (in executeTurn)
  - Save on game over (resignation/elimination)
  - Save on disconnect timeout (before removing game from memory)
  - resumeGame(id) — loads saved game into memory with both players as null (disconnected)
  - getSavedGames() — delegates to db.listGames()
  - deleteSavedGame(id) — delegates to db.deleteGame()
- Updated server index.ts:
  - Instantiates GameDatabase, passes to GameManager
  - GET /api/games now returns { active, saved }
  - POST /api/games/:id/resume — reload game from DB for WebSocket reconnection
  - DELETE /api/games/:id — remove saved game
