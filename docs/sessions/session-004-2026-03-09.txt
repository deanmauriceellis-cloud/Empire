## Decisions & Rationale
- Decision: Single game.ts file for all core mechanics — Rationale: Functions are heavily
  interdependent (killUnit calls scan, moveUnit calls embark, attackUnit calls killUnit+moveUnit).
  Splitting would create circular dependencies.
- Decision: Mutable GameState pattern — Rationale: Matches the original C code's mutation style.
  Simpler than immutable updates for a game engine. State is always passed explicitly.
- Decision: Added rngState to GameState — Rationale: Combat and satellite directions need
  randomness. Storing the PRNG state in GameState ensures determinism and serializability.
- Decision: City lookup by ID (find) rather than array index — Rationale: City IDs don't
  necessarily match array indices, especially after game state changes.
- Decision: Endgame check skips when both players have 0 total — Rationale: Prevents false
  elimination triggers in early game or test scenarios before units are produced.
