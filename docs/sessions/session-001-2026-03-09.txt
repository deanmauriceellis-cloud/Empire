## Decisions & Rationale
- Decision: Replace Colyseus with plain ws — Rationale: Empire is turn-based, doesn't need real-time delta sync or room management
- Decision: Replace Redis with in-memory Map — Rationale: single server process, turn-based game, active state fits in memory
- Decision: Replace PostgreSQL with SQLite — Rationale: simpler deployment, no external DB service, sufficient for game saves
- Decision: Remove Better Auth — Rationale: premature; can add OAuth later without architectural changes
- Decision: Remove Turborepo — Rationale: 3-package monorepo doesn't benefit from build caching
- Decision: Remove tsup — Rationale: shared package can be consumed directly via workspace references
- Decision: Single-server deploy — Rationale: user self-hosts, one Node process serves everything

## Full Conversation Summary
Session began with the standard protocol: loaded STATE.md, CHANGELOG.md, and session-protocol.md, ran drift check (no drift found), confirmed all files under archive thresholds. Briefed user on Phase 0 status and suggested starting scaffolding.

User redirected the session toward architecture review, asking to simplify the stack. They noted they host their own website and are fine using Node.js for multiplayer state management. This signaled that the original plan's multi-service cloud architecture (Cloudflare Pages + Fly.io + R2, Colyseus, Redis, Better Auth) was over-engineered for their use case.

I presented a detailed analysis of what was over-engineered and why, proposing replacements: plain WebSockets instead of Colyseus, in-memory state instead of Redis, SQLite instead of PostgreSQL, simple sessions instead of Better Auth, pnpm workspaces instead of Turborepo, and a single Docker deploy instead of 3 services. Emphasized that all game logic (Phases 1-4) and all rendering/UI (Phases 7-8, 10) would be completely unaffected.

User asked whether the simplification would constrain future game logic and graphical development. Confirmed it would not — the three-package separation (shared/client/server) keeps concerns cleanly isolated. The game engine doesn't know about networking, the renderer doesn't know about the server, and switching from plain WebSockets to something heavier later would only require changing the transport layer.

User approved and asked to rewrite the plan. PLAN.md was rewritten with the simplified stack, reducing total steps from 55 to 48 while preserving all gameplay and graphics phases. Session ended with documentation updates.
